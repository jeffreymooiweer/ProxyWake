# ProxyWake v4.2.4

## Summary

Stability release. Every feature was exercised end-to-end against a production-style deployment (2 gunicorn workers, fresh data directory), which surfaced a set of long-standing bugs — several of them breaking core functionality in default Docker installs. All are fixed in this release, and device status checks got noticeably faster.

## Fixed

### Critical

- **Stored SSH/IPMI credentials could never be decrypted** when `PROXYWAKE_SECRET_KEY` was not set: every call generated a fresh random key. The secret key is now generated once and persisted in the data directory (`secret.key`, mode 600). Existing credentials that were saved under the old behaviour are unrecoverable and must be re-entered once.
- **Login sessions broke between gunicorn workers** for the same reason — roughly half of all requests returned 401 after login. Sessions now survive worker switches and container restarts.
- **First boot with a fresh database could crash the server**: both workers raced in schema creation and one failed on `table app_setting already exists`. Schema creation now runs under a file lock.
- **Deleting a device that had wake history failed with HTTP 500** (`NOT NULL constraint failed: wake_event.device_id`). Wake events, schedules and dependency links are now removed together with the device.

### Reliability

- Verified-wake job status is stored in the database instead of per-process memory, so the wake progress UI works regardless of which worker answers the poll
- Scheduled wakes run in exactly one worker (file lock) and align to minute boundaries, so schedules can no longer be skipped or double-fired
- The waiting page fires the wake once (non-blocking) instead of starting a new blocking wake every 3 seconds, which could exhaust all server threads
- Wake progress polling on the Devices page recovers from transient errors instead of leaving the spinner stuck

### Correctness

- Log level filtering on the Logs page never matched because the log timestamp contains colons
- Editing a device no longer overwrites webhook URLs with their masked (`***`) display value; full backups export the real webhook URLs so restores round-trip
- Deleting a group or NPM host clears the reference on linked devices
- Schedules validate that the device exists and that hour/minute are in range; the API returns the configured `days`
- The `?host=` query parameter of `/api/wake/by-host` was unreachable because the always-present `Host` header took precedence

## Changed

- `POST /api/public/wake/{domain}` accepts `?wait=false` to send a wake without blocking until the device is online
- Device status checks (`/api/devices?status=true`, `/api/stats`) run concurrently instead of serially, so device lists stay fast with several offline devices

## Upgrade

```bash
docker pull jeffersonmouze/proxywake:4.2.4
```

Recreate the container with the same appdata path and environment variables.

**Note:** if you saved SSH or IPMI passwords in an earlier version *without* setting `PROXYWAKE_SECRET_KEY`, re-enter them once after upgrading — the old values were encrypted with throwaway keys and cannot be recovered.
