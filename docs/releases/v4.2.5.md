# ProxyWake v4.2.5

## Summary

Working Traefik and Caddy integrations plus a security fix for API-key brute forcing. The old Traefik/Caddy snippets on the Integration page were examples that could not work as-is; they are replaced by tested, plugin-free configurations built on each proxy's native forward-auth mechanism.

## Fixed

- **API-key brute force was never throttled.** Rate-limit decorators sat inside the auth decorators, so rejected (401) requests were returned before the limiter ever counted them — invalid keys could be tried without limit. The limiter now wraps auth on `/api/wake/by-host`, device wake and network scan: invalid keys are throttled (60/min on by-host), valid traffic is unaffected.
- `/api/wake/by-host` no longer writes a skipped wake event for every proxied request when the device is already online. On busy proxies (NPM mirror, forward-auth) this flooded the wake-event table.

## Changed

- **Traefik snippet rewritten** — now uses the built-in `forwardAuth` middleware, no plugins:

  ```yaml
  labels:
    - "traefik.http.middlewares.proxywake.forwardauth.address=http://proxywake:5001/api/wake/by-host?api_key=KEY"
    - "traefik.http.routers.my-service.middlewares=proxywake"
  ```

- **Caddy snippet rewritten** — built-in `forward_auth` plus `handle_errors` (Caddy 2.8+) redirecting to the waiting page while the device boots:

  ```caddyfile
  app.example.com {
      forward_auth http://proxywake:5001 {
          uri /api/wake/by-host?api_key=KEY
      }
      reverse_proxy 192.168.1.50:8080
      handle_errors 502 504 {
          redir * http://proxywake:5001/waiting?domain={host}
      }
  }
  ```

- The API key may be passed as an `?api_key=` query parameter — forward-auth middlewares cannot attach custom headers. Header keys still take precedence. Use an internal (LAN/Docker) ProxyWake URL so the key never crosses a public network.
- Requests with a **valid** API key are exempt from the `/api/wake/by-host` rate limit, since forward-auth setups route every proxied request through it.
- `docs/examples/traefik.md` and `docs/examples/caddy.md` rewritten around the new configurations, including behaviour notes (e.g. forward-auth blocks requests if ProxyWake itself is down).

## Upgrade

```bash
docker pull jeffersonmouze/proxywake:4.2.5
```

Recreate the container with the same appdata path and environment variables. If you used the old Traefik/Caddy snippets, replace them with the newly generated ones from **Integration → Traefik / Caddy**.
