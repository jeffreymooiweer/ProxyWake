# ProxyWake v4.2.6

## Summary

Hotfix for a first-boot race in secret-key creation, found while verifying the Nginx Proxy Manager integration end-to-end against a real nginx.

## Fixed

- **Secret-key creation race on first boot.** With two gunicorn workers starting on a fresh data directory, one worker could read `secret.key` in the window before the other finished writing it — leaving it with an empty key and, once the empty file existed, every subsequent boot broken too (HTTP 500 on login/setup, unusable sessions until the file was deleted by hand). Key creation is now serialized with a file lock, and an existing empty `secret.key` is repaired automatically on the next start.

Affects 4.2.4 and 4.2.5 installs that boot two workers on an empty data directory without `PROXYWAKE_SECRET_KEY` set. If you hit this, upgrading is enough — the empty file is replaced on startup.

## Verified

The generated **Nginx Proxy Manager snippets** were exercised end-to-end against a real nginx 1.24 (the same line NPM ships):

- `nginx -t` accepts both generated snippets verbatim
- A visit while the device sleeps fires the `mirror` trigger → `/api/wake/by-host` → wake sent, and the `error_page` rule answers `302` to `/waiting?domain=<host>`
- After the device comes up, the same URL serves the backend normally
- 15 requests while online produced zero additional wake events (no table flood) and no interference with proxied traffic

## Upgrade

```bash
docker pull jeffersonmouze/proxywake:4.2.6
```

Recreate the container with the same appdata path and environment variables.
